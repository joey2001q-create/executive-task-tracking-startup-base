# 状态同步规则（任务巡检 + 自动更新）

每天定时跑一次，**巡检多维表里所有未完成任务**，**直接拉任务负责人的原始数据（群聊/单聊/日程/妙记/邮箱/任务）来判断任务是否完成或有新进展**，状态变化就更新任务表 + 追加跟进记录。

## 核心：基于负责人原始数据判断（不是基于报告）

与"报告驱动"不同，巡检是**直接拉负责人的多维原始数据**，由 AI 判断每条任务的真实进展，更准、更及时。

## 同步流程

```
[定时 cron 触发，默认北京时间每天 21:00，时区 Asia/Shanghai]
   ↓
1. 拉多维表所有"未完成"任务（状态 ≠ 已完成/已取消）
   base +record-list {{TABLE_ID_任务信息表}}，过滤状态
   ↓
2. 按负责人分组，每位负责人拉近 N 天（默认7）的原始数据：
   用外部 token 管理能力以 <负责人 open_id> 身份拉取：
   - 群聊   im +messages-search --sender <open_id> --chat-type group
   - 单聊   im +messages-search --sender <open_id> --chat-type p2p
   - 日程   calendar +agenda
   - 妙记   minutes +search --owner-ids <open_id>
   - 任务   task +get-my-tasks（负责人自己的飞书原生任务状态）
   - 邮箱   mail（如已授权，拉标题/摘要）
   ↓
3. 对每条未完成任务，AI 在该负责人原始数据中检索相关证据：
   - 匹配：任务标题关键词 / 客户名 / 项目名 出现在哪条消息/日程/妙记/任务里
   - 完成信号（"已完成""搞定""上线了""已提交""关闭"）→ 判定已完成
   - 进展信号（"在做""推进中""刚开了会"）→ 进行中 + 记进度
   - 卡点信号（"在等""被阻塞""没资源"）→ 阻塞 + 记卡点
   - 完全没证据 + 超过截止日 → 已逾期
   - 完全没证据 + 7 天未提及 → 疑似失联
   ↓
4. 状态有变化 → base +record-upsert --record-id 更新任务表状态
   同时 base +record-upsert 追加一条跟进记录（{{TABLE_ID_任务跟进记录表}}），
   追进内容写明：旧状态→新状态 + 判断依据（证据摘要 + 源消息 ID）+ 巡检时间
   ↓
5. 输出巡检报告（见下），推送汇总群
```

## 与任务匹配的证据引用

每条状态更新都要带证据，便于老板复核：
```
chat:oc_xxx:om_xxx     群聊/单聊证据
calendar:event_xxx     日程证据
minutes:minute_xxx     妙记证据
task:task_guid         负责人原生任务状态
```

## 状态变化判断矩阵

| 多维表当前状态 | 报告中状态 | 动作 | 跟进记录 |
|---|---|---|---|
| 待执行 | 待执行 | 无变化 | 不追加 |
| 待执行 | 进行中 | 更新为"进行中" | ✅ 追加：状态变更 |
| 待执行 | 已完成 | 更新为"已完成" | ✅ 追加：完成确认 |
| 进行中 | 进行中 | 无变化 | 看卡点/进度是否变，是则追加 |
| 进行中 | 已完成 | 更新为"已完成" | ✅ 追加：完成 |
| 进行中 | 阻塞 | 更新为"阻塞" | ✅ 追加：阻塞告警 |
| 进行中 | （报告中未提到）| 检查截止日 | 过期 → 更新为"已逾期"+追加 |
| 阻塞 | 阻塞 | 无变化 | 看卡点是否变，是则追加 |
| 阻塞 | 进行中 | 更新为"进行中" | ✅ 追加：解除阻塞 |
| 阻塞 | 已完成 | 更新为"已完成" | ✅ 追加：完成 |
| 已完成 | （任何）| 不动 | 不追加（已完成任务不更新）|
| 已逾期 | 已完成 | 更新为"已完成" | ✅ 追加：完成 |
| 已逾期 | 进行中 | 仍标"已逾期" | 追加：进度更新 |

## 自动"已逾期"判定

```python
def check_overdue(task):
    if task.status in ("进行中", "阻塞"):
        if task.due_date and task.due_date < today():
            return True
    return False
```

执行时机：每次同步时，对所有未完成任务跑一遍。

## 自动"疑似失联"判定

任务在多维表里"进行中"或"阻塞"，但**最近 7 天**的报告中**完全没被提到**：

```
⚠️ 疑似失联（7 天未提及）
任务: 【张三】客户A合同谈判
负责人: 张三
最后提及: 6/04
当前状态: 阻塞
```

→ 追加跟进记录提示老板/老板/张三 关注。

## 何时不动

- **已完成任务**（状态=已完成）：完全不动，避免"事后修史"
- **状态没变 + 报告也没提**：不追加记录（避免噪音）
- **报告里信息不足**（无明确动作）：标记"待复核"，不自动改状态

## 安装时创建的定时巡检 cron

安装 `skills-package-v6` 时由 Agent 创建 cron job。不要在 skill 包里写死 job id。

- **名称**：任务巡检与状态同步
- **job id**：创建成功后记录实际生成的 `TASK_TRACKING_CRON_JOB_ID`
- **调度**：北京时间每天 21:00 (Asia/Shanghai)
- **会话**：isolated
- **推送**：黄杰（user:{{BOSS_OPEN_ID}}）或管理员指定目标
- **逻辑**：拉未完成任务 → 按负责人拉原始数据 → AI 判断进展 → 状态变化则更新任务表+追加跟进记录 → 发巡检报告

如需调整时间/推送目标，用实际生成的 `TASK_TRACKING_CRON_JOB_ID` 更新 cron。

## 老板配置项

```yaml
sync:
  # 状态变更时是否自动更新（默认 true）
  auto_update_status: true
  
  # 是否自动标记"已逾期"（默认 true）
  auto_mark_overdue: true
  
  # 多久未提及触发"疑似失联"（默认 7 天）
  missing_alert_days: 7
  
  # 每次最多同步多少条（避免长任务）
  max_sync_per_run: 100
  
  # 同步报告推送目标
  report_push_to: "chat:{{REPORT_CHAT_ID}}"
```

## 失败处理

| 失败类型 | 重试策略 |
|---|---|
| 拉取多维表失败 | 重试 3 次，间隔 1s/3s/5s |
| 拉取高管数据失败 | 跳过该高管，继续其他 |
| 单条任务更新失败 | 跳过该条，继续其他，统计中报告 |
| 跟进记录创建失败 | 任务已更新，跟进记录失败 → 警告，不回滚 |

## 性能

- 拉取多维表全量 + 解析 12 条任务：~2s
- 拉高管数据（3 人）+ 解析：~30s（依赖 feishu-executive-tracker）
- 对比 + 更新 + 跟进：~10s
- 总耗时：~45s（10 人以内）

## 重要：避免误操作

**首次运行建议 dry_run 模式**：

```yaml
sync:
  dry_run: true  # 只报告，不写入
```

老板看完"将要做什么"清单后，再切到 `dry_run: false` 真正执行。
