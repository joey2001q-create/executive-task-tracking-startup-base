# 任务追踪与巡检系统

> **唯一数据来源**：黄杰在对话框中说"给XX布置个任务...""让XX做..."等指令。
> 每日 21:00 自动巡检所有未完成任务，获取负责人6类数据，判断状态并追加跟进记录。

## ⚠️ 强制触发场景（铁律，必须执行）

### 触发一：老板布置任务 → 必入表
只要黄杰说"给我自己布置个任务""给 XX 布置任务""让 XX 做 Y""告诉 XX 截止前完成 W"等（不管对象是黄杰本人还是其他人），**一律走本 skill 写入多维表**，而不是只建飞书原生任务。默认状态"待执行"，描述记"布置人:黄杰+原话+源消息+【老板布置任务 - 黄杰】"。

### 触发二：每日定时巡检 → 自动更新任务状态
每天 21:00（已配 cron job `49e916e3-ca91-41f9-bfb2-ce5fe8868836`）巡检多维表所有未完成任务：
1. 用 `feishu-data-collector-v3` 获取任务负责人的6类数据（群聊/单聊/日程/任务/妙记/邮箱）
2. AI 分析数据，判断任务是否完成或有新进展
3. 如有进展，添加跟进记录；如明确完成，更新状态为"已完成"
4. 生成巡检报告（L1 简报 + L2 详情）

**关键约束：**
- 飞书原生任务（`lark-cli task +create`）可以建（方便提醒），但**不能替代**多维表写入——两者都要做，多维表是台账核心。
- 写入用 `lark-cli base +record-upsert`，bot 身份。
- **link 字段（关联任务）格式**：直接传 record_id 数组 `["recXXX"]`，不要用 `{table_id,record_ids}` 对象（会报 800010701）。
- **select/text/user 字段**：状态用纯字符串、负责人用 `[{"id":"ou_xxx"}]`、URL/text 用纯字符串。
- **去重铁律**：写入前必须先查现有任务/跟进记录，避免重复创建。

## 适用场景

- **老板布置任务**：黄杰说"给张三布置个任务：6/15 前完成产品 V2 评审" → 自动入库
- **每日自动巡检**：每天 21:00 自动巡检，获取负责人数据，判断状态，追加跟进记录
- **状态闭环**：任务负责人在群聊/单聊中提及任务进展 → 自动添加跟进记录

## 工作流程

### Step 1：接收输入

**唯一输入模式：老板直接布置**

黄杰在对话框中说"给 XX 布置个任务：XXX"，agent 解析出"任务标题/负责人/截止/描述"后入库。

#### 解析规则

老板的指令格式（自然语言，宽松匹配）：

| 老板说 | 解析结果 |
|---|---|
| "给张三布置个任务：6/15 前完成产品 V2 评审" | 任务: 产品 V2 评审 / 负责人: 张三 / 截止: 6/15 / 状态: 待执行 |
| "让李四下周给客户王总出方案" | 任务: 给客户王总出方案 / 负责人: 李四 / 截止: 下周五 / 状态: 待执行 |
| "安排王五去跟客户赵总对接" | 任务: 跟客户赵总对接 / 负责人: 王五 / 截止: — / 状态: 待执行 |
| "告诉赵六，6/20 前必须把合同签了" | 任务: 签合同 / 负责人: 赵六 / 截止: 6/20 / 状态: 待执行 |
| "麻烦你通知小张处理下 XXX 报销" | 任务: 处理 XXX 报销 / 负责人: 小张 / 截止: — / 状态: 待执行 |

**关键解析逻辑**：
1. **找负责人**：老板话里提到的人名（去飞书通讯录查 open_id），或在飞书消息上下文里被 @ 的人
2. **找截止**：日期关键词（今天/明天/下周/几月几日/尽快），"尽快"需追问具体日期
3. **找任务标题**：动词开头 + 可验收结果（如"出方案""完成 V2 评审""签合同"），没有明确动词则用整句概括
4. **状态**：默认"待执行"（还没开始）
5. **创建人**：自动为 bot 调 API（系统字段）；**详细描述里写明"由黄杰布置，{原话}"**

**反例**（不录入）：
- ❌ "张三今天挺忙" → 不是任务
- ❌ "下周一开个会" → 这是日程不是任务（用 `lark-cli calendar` 建日程，不入库）
- ❌ 老板转发客户消息但没指示"让 XX 处理" → 反问老板"这个要布置给谁？"

**有歧义时**（负责人/截止说不清）→ **反问老板一次**，再写。

详见 [config-schema.md](references/config-schema.md) 的 `input.source: command` 段落。

## 工作流程

### 场景一：老板布置任务

#### Step 1：解析指令

从黄杰的自然语言指令中提取：

| 字段 | 提取规则 |
|---|---|
| 任务标题 | 动词开头 + 可验收结果（如"完成 V2 评审""出方案"）|
| 负责人 | 指令中提到的人名 → `feishu_search_user` 查 open_id |
| 截止日期 | 日期关键词（今天/明天/下周/几月几日/尽快）|
| 详细描述 | "布置人:黄杰+原话+源消息" |
| 状态 | 默认"待执行" |

**反例**（不入库）：
- ❌ "张三今天挺忙" → 不是任务
- ❌ "下周一开个会" → 这是日程
- ❌ 老板转发客户消息但没指示"让 XX 处理" → 反问"这个要布置给谁？"

#### Step 2：去重检查（铁律）

查询任务信息表 `{{TABLE_ID_任务信息表}}`，检查是否已存在相同任务：

```bash
lark-cli --profile {{APP_ID}} --as bot base +record-list \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id {{TABLE_ID_任务信息表}} \
  --format json 2>/dev/null
```

去重规则：
1. **精确匹配**：`任务标题 + 负责人 open_id` 完全相同 → 已存在，不创建
2. **模糊匹配**：标题相似度 > 80% + 负责人相同 → 已存在，不创建
3. **新任务**：以上都不匹配 → 创建

#### Step 3：创建任务

```bash
lark-cli base +record-upsert \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id {{TABLE_ID_任务信息表}} \
  --json '{
    "任务标题": "产品 V2 评审",
    "详细描述": "布置人: 黄杰 ({{BOSS_OPEN_ID}})\n原话: 给张三布置个任务：6/15 前完成产品 V2 评审\n布置时间: 2026-06-21",
    "负责人": [{"id": "ou_张三_open_id"}],
    "任务状态": "待执行",
    "是否卡点": "否"
  }' \
  --as bot \
  --profile {{APP_ID}}
```

#### Step 4：添加初始跟进记录

创建任务后，在跟进记录表添加一条"任务创建"记录：

```bash
lark-cli base +record-upsert \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id {{TABLE_ID_任务跟进记录表}} \
  --json '{
    "关联任务": ["rec_xxx"],
    "跟进内容": "【任务创建】由黄杰布置，状态：待执行",
    "跟进人": [{"id": "{{BOSS_OPEN_ID}}"}]
  }' \
  --as bot \
  --profile {{APP_ID}}
```

### 场景二：每日定时巡检

#### Step 1：拉取未完成任务

从任务信息表拉取所有状态 ≠ "已完成" 的任务：

```bash
lark-cli --profile {{APP_ID}} --as bot base +record-list \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id {{TABLE_ID_任务信息表}} \
  --format json 2>/dev/null
```

#### Step 2：获取负责人6类数据

对每个未完成任务的责任人，使用 `feishu-data-collector-v3` 采集过去7天数据：

```bash
python3 feishu-data-collector-v3 /tmp/巡检_<姓名>.json '<open_id>:<姓名>'
```

采集6类数据：日程、任务、妙记、群聊、单聊、邮箱。

**前提**：负责人已完成 OAuth 授权，token 存入多维表。

#### Step 3：AI 分析判断状态

分析负责人的6类数据，判断每个任务的状态：

| 状态 | 判断依据 |
|------|---------|
| **已完成** | 聊天记录/日程/任务中明确提及"已完成""搞定了""交付了"等 |
| **有进展** | 数据中出现任务关键词，有实质性推进但未完成 |
| **阻塞** | 数据中出现"卡住了""等XX""没资源"等 |
| **无进展** | 7天内数据中未出现任务相关关键词 |

**分析方法**：
1. 从任务标题提取关键词（如"产品 V2 评审" → 关键词："V2""评审""产品"）
2. 在6类数据中搜索这些关键词
3. 匹配到的内容作为证据，判断状态

#### Step 4：更新任务状态

如判断为"已完成"：
```bash
lark-cli base +record-upsert \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id {{TABLE_ID_任务信息表}} \
  --record-id "rec_xxx" \
  --json '{"任务状态": "已完成"}' \
  --as bot --profile {{APP_ID}}
```

#### Step 5：追加跟进记录（去重）

**铁律：追加前必须先查现有跟进记录，避免重复。**

```bash
# 查该任务已有的跟进记录
lark-cli --profile {{APP_ID}} --as bot base +record-list \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id {{TABLE_ID_任务跟进记录表}} \
  --format json 2>/dev/null
```

如跟进内容已存在（相同或高度相似）→ 不创建。

如为新跟进 → 创建：
```bash
lark-cli base +record-upsert \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id {{TABLE_ID_任务跟进记录表}} \
  --json '{
    "关联任务": ["rec_xxx"],
    "跟进内容": "【进展】从群聊中发现：已开始准备评审材料（证据：msg_xxx）",
    "跟进人": [{"id": "ou_xxx"}]
  }' \
  --as bot --profile {{APP_ID}}
```

#### Step 6：生成巡检报告

**L1 简报（飞书 IM）**：

简报只体现三类任务：**有进展**、**已逾期**、**有风险**。其他内容不体现。

```
**📋 任务巡检报告（YYYY-MM-DD）**

**🔄 有进展（X条）**
- 【负责人】任务标题 | 简要说明

**⚠️ 已逾期（X条）**
- 【负责人】任务标题 | 逾期天数 | 简要说明

**🔴 有风险（X条）**
- 【负责人】任务标题 | 风险说明

**📄 详情** <云文档链接>
```

**L2 详情（云文档）**：
- 使用 `lark-cli docs +create --api-version v2 --doc-format markdown` 创建
- 内容包含：
  - 巡检概览统计
  - 每个任务的详细分析
  - 证据来源（消息ID、日程标题、任务状态等）
  - 关键发现
  - 跟进记录时间线
  - 数据覆盖说明

#### Step 7：存储巡检报告

创建云文档后，将链接写入巡检报告存储表：

```bash
# 创建云文档
lark-cli docs +create \
  --api-version v2 \
  --doc-format markdown \
  --content "$(cat /tmp/巡检报告.md)" \
  --as bot \
  --profile {{APP_ID}}

# 将链接写入巡检报告存储表
lark-cli base +record-upsert \
  --base-token {{AGENT_BASE_TOKEN}} \
  --table-id tblpDEjZJAmc6wJD \
  --json '{
    "巡检日期": "2026-06-21",
    "巡检报告": "https://ldkj.feishu.cn/docx/xxx"
  }' \
  --as bot \
  --profile {{APP_ID}}
```

## 去重规则（铁律）

### 任务去重
创建任务前，必须检查任务信息表：
1. **精确匹配**：`任务标题 + 负责人` 完全相同 → 跳过
2. **模糊匹配**：标题相似度 > 80% + 负责人相同 → 跳过

### 跟进记录去重
创建跟进记录前，必须检查该任务已有的跟进记录：
1. **内容匹配**：跟进内容完全相同 → 跳过
2. **时间匹配**：同一天内相同任务 + 相似内容 → 跳过

## 异常处理

| 异常 | 行为 |
|---|---|
| 负责人未授权 | 标记"数据不可用"，任务状态不变，报告中说明 |
| 多维表权限拒绝 | 检查 bot scope（`bitable:app:readonly` + `bitable:app:write`）|
| 跟进记录重复 | 先查后写，重复则跳过 |
| token 过期 | 通知负责人重新授权 |
| 数据获取失败 | 单条失败不影响其他任务，记录错误原因 |

## 报告发送规则（防重复）

**铁律：报告只发一次！**

❌ **禁止行为**：
- 先发送详细报告（post 类型），再发总结卡片（interactive 类型）
- 用 `message` 工具发送后，再用文本回复重复发送

✅ **正确做法**：
- 只选择一种方式发送报告
- 推荐：使用 `message` 工具发送完整的 L1 简报 + L2 云文档链接
- 不额外发送总结性文本回复

## 安全与合规

- **写入前必须先查重**：任务/跟进记录都要先查现有记录
- **不修改已完成任务的核心字段**（标题/创建时间）→ 只更新状态/卡点
- **删除任务不自动做** → 误识别可恢复
- **每次同步记录到跟进表** → 完整审计日志
- **个人数据脱敏**：分析时排除个人隐私内容
