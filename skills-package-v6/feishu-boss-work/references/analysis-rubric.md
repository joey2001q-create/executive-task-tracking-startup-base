# 高管洞察分析流程

用飞书协作数据回答老板关心的四个问题：

- 关键事项：该高管在指定时间真正推动了什么。
- 关键决策：该高管做了哪些会影响业务方向、资源、客户、团队或风险的决定。
- 重大风险：有哪些延期、客户、财务、组织、法务、舆情或执行风险正在出现。
- 需老板介入：哪些事项超出高管权限，或需要老板拍板、协调、背书或升级。

同时输出一个高管管理画像，帮助老板快速理解该高管本周期呈现出的管理风格、优势和需要继续观察的短板。

## 1. 定义分析边界

必须先明确：

- 分析对象：高管名单、职务、飞书 ID、邮箱、部门。
- 时间范围：日、周、月或自定义时间段。
- 公司重点：OKR、重点项目、重点客户、预算/回款目标、产品发布、组织调整、风险主题。
- 数据范围：聊天、日程、妙记/会议纪要、任务、待办、邮件。
- 授权范围：管理员授权、用户授权、审计流程、法务/合规批准、可见范围限制。

## 2. 数据源角色

按来源理解业务含义，不要把所有数据等价处理。

| 来源 | 主要价值 | 常见信号 |
| --- | --- | --- |
| 聊天记录 | 真实推进、临时卡点、跨部门协调、风险暴露 | 催办、拍板、升级、争议、负责人、截止时间 |
| 日程 | 时间投入和管理关注点 | 客户会、复盘会、1:1、评审会、事故会、经营会 |
| 妙记/纪要 | 决策过程、会议结论、行动项、分歧 | 会议总结、逐字稿、章节、待办、决策、未决问题 |
| 任务/待办 | 执行落地和责任闭环 | 负责人、状态、截止时间、延期、完成、阻塞 |
| 邮件 | 正式承诺、高敏事项、外部沟通 | 合同、报价、客户承诺、法务、预算、招聘、回款 |

## 3. 标准化记录

把原始数据转成统一记录，再分析：

```json
{
  "source_type": "chat",
  "timestamp_start": "2026-06-17T10:00:00+08:00",
  "executive_ids": ["exec_001"],
  "title": "重点客户续约折扣讨论",
  "summary": "CRO 要求销售、法务和财务在客户会前对齐折扣边界。",
  "business_entities": {
    "projects": ["企业续约"],
    "customers": ["某重点客户"],
    "departments": ["销售", "法务", "财务"]
  },
  "activity_tags": ["key_matter", "decision", "risk"],
  "evidence": {
    "source_ids": ["chat:oc_xxx:om_xxx"],
    "quote": null
  },
  "confidence": "high"
}
```

## 4. 识别关键事项

判断一个事项是否关键，看这些条件：

- 关联公司 OKR、战略项目、重点客户、产品发布、回款、成本或组织建设。
- 反复出现在聊天、日程、妙记、任务或邮件中。
- 高管亲自推动、拍板、协调或承担责任。
- 有负责人、截止时间、里程碑、依赖或业务结果。
- 涉及多个部门、关键客户、预算、交付或组织稳定。

输出格式：

```text
事项：某重点客户续约
高管动作：协调销售、法务、财务确认折扣边界
当前状态：客户会前准备阶段
业务含义：影响本季度续约收入
证据：chat:..., calendar:...
置信度：high
```

## 5. 识别关键决策

决策信号通常来自聊天、妙记、会议纪要和邮件。

重点识别：

- 批准预算、价格、折扣、编制、招聘或政策例外。
- 确认方案、路线、优先级、上线/延期、暂停/取消项目。
- 解决跨部门分歧。
- 升级或降级客户问题。
- 明确“谁负责、何时完成、按什么标准完成”。

输出时要包含：

- 决策内容。
- 决策原因或背景。
- 影响范围。
- 后续责任人和时间。
- 证据 ID。
- 置信度。

## 6. 识别重大风险

风险不能只看“风险”两个字，也要看异常行为和缺口。

常见风险：

- 客户风险：投诉、流失、续约失败、关键客户升级。
- 收入/财务风险：回款逾期、预算超支、毛利下降。
- 交付风险：延期、质量问题、依赖未解决。
- 组织风险：关键人离职、招聘失败、团队冲突。
- 合规/法务风险：合同争议、数据安全、监管问题。
- 声誉风险：舆情、公关、外部投诉。

高风险信号：

- 有截止时间但没有负责人。
- 多次讨论但没有结论。
- 被频繁升级或反复催办。
- 涉及关键客户、合同、资金、监管或核心员工。
- 任务状态显示延期，但聊天/日程中没有补救动作。

## 7. 判断是否需老板介入

满足任一条件，通常需要提示老板：

- 超出高管授权范围，需要老板拍板。
- 需要跨高管、跨部门或组织层面的资源协调。
- 涉及关键客户、重大合同、重大回款或关键候选人。
- 存在法务、合规、声誉或重大经营风险。
- 高管已经推动但卡在资源、权限、政策或冲突上。
- 延迟处理会造成明显业务损失。

输出格式：

```text
事项：重点客户续约折扣边界
为什么需要老板：折扣超出 CRO 常规授权，且影响本季度收入
建议动作：老板确认价格底线，必要时参与客户会
时间敏感性：48 小时内
证据：chat:..., mail:...
置信度：high
```

## 8. 交叉验证

不要只依赖单一来源做强结论。

优先级：

- 高置信：妙记/会议纪要或邮件中有明确结论，且聊天/任务有后续动作。
- 中置信：多个来源指向同一主题，但缺少正式决策文本。
- 低置信：只有单条聊天、单个日程标题或不完整任务信息。

## 9. 输出分层

老板简报：

- 只写高管工作主线、卡点风险、关键决策、需老板介入。
- 每位高管加入一句管理画像，例如“本周期呈现出客户攻坚型，主要围绕重点客户续约推动跨部门对齐；需要继续观察是否能把临时协调沉淀为机制”。
- 不展示原始聊天、任务数量、消息数量、邮件数量或证据长列表。

飞书云文档详情：

- 放具体工作内容、任务、风险、证据 ID、来源覆盖和数据缺口。
- 放管理画像的依据、置信度、横向对比线索和需要继续观察的短板。
- 适合复盘、追责、审计和进一步分析。

## 10. 高管画像

画像类型建议：

- 战略推进型：推动关键目标、战略项目、重大资源配置。
- 组织管理型：重视团队建设、干部培养、机制和问责。
- 风险响应型：善于发现和处理风险，但可能偏被动响应。
- 客户攻坚型：聚焦关键客户、续约、交付、投诉和一线反馈。
- 协同枢纽型：能连接多个部门解决复杂问题。
- 执行闭环型：重视负责人、截止时间、跟进和复盘。
- 救火过载型：大量处理紧急问题，可能缺少前置规划和机制。
- 需关注型：关键事项、决策、风险或闭环存在明显短板。

画像要写成阶段性判断：

```text
本周期呈现出[画像类型]，主要依据是[证据摘要]；需要继续观察[短板或风险]。[置信度]
```

## 11. 判断原则

- 不把“忙”当成“有效”。
- 不把“消息多”当成“贡献大”。
- 不把“没有证据”写成“没有做”。
- 不把阶段性画像写成永久性人格评价。
- 对推断使用“可见数据表明”。
- 对缺失来源明确标注数据缺口。
- 对个人隐私、法律特权和无关私人内容默认省略或脱敏。
